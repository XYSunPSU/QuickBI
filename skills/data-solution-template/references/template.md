# 数据方案模板：0330 | 孕产单店模型数据方案

来源模板：飞书文档「0330 | 孕产单店模型数据方案」

用途：以后创建数据方案时，默认参考这份结构与写法。方案要从业务目标出发，落到口径、链路、模型表、看板呈现和待确认风险，不只写技术实现。

## 标准结构

### 一、背景与目标

用一段话说明：

- 业务方、财务方、技术方要共同解决什么经营问题。
- 要统一哪些口径、归集哪些数据。
- 最终产物是什么，例如模型表、数据看板、监控或诊断工具。
- 产物支撑的业务动作，例如经营诊断、费用优化、门店盈利分析。

推荐句式：

> 联合{业务方/财务方/相关部门}梳理{收入/成本/费用/业务过程}明细，统一{归集口径/统计口径/分摊规则}，建设{主题模型/看板/数据产品}。实现{核心对象}的{指标/状态/盈利/效率}实时监控，支撑{经营诊断/优化决策/过程管理}。

### 二、需求文档

放需求来源或链接。若暂时没有正式需求文档，也保留章节，后续补充。

应包含：

- 原始需求文档链接。
- 业务沟通纪要或会议链接。
- 需求边界说明，例如统计周期、统计对象、上线范围。

### 三、核心成员与项目群组

使用三列表格：

| 核心成员 | 部门 | 项目角色与负责事项 |
| --- | --- | --- |
| {成员} | {部门} | {负责事项} |

填写重点：

- 业务 owner：定义业务口径、验收看板。
- 财务 owner：确认收入、成本、费用归集和分摊规则。
- 技术 owner：负责数据接入、建模、看板交付。
- 运营或一线代表：确认看板使用方式和异常解释。

### 四、解决方案

解决方案固定拆成 5 个小节。

#### 1、数据大图

放公司或业务域已有的数据大图链接，并说明本方案位于哪条业务链路。

要写清：

- 上游系统：如 ERP、POS、审批、企微、财务报表、人力报表等。
- 中间处理：如清洗、映射、归集、分摊、口径转换。
- 下游产物：如 ODS/DWD/ADS 表、BI 看板、经营分析报表。

#### 2、数据口径

这是模板的核心部分。用表格逐项列出口径，不要只写指标名。

标准列：

| 分类 | 费用项/指标项 | 当前口径/说明 | 主要数据来源 | 备注/风险 |
| --- | --- | --- | --- | --- |
| {一级分类} | {明细项} | {统计口径、确认规则、分摊规则} | {系统/报表/审批/人工提供方} | {待确认、风险、技术支持点} |

写法要求：

- 分类可以合并展示，例如人员费用项、日常管理要项、运营费用。
- 每个费用项都要说明业务含义和确认口径。
- 数据来源要具体到系统、审批流、报表、负责人或人工环节。
- 备注/风险要标注是否需要技术支持、业务确认、财务人工识别或规则补充。
- 暂无数据或暂不纳入的项目可以保留并标注删除线或“暂无”，方便后续复盘范围变化。

模板里的典型分类：

- 运营利润：销售额、产品成本、毛利。
- 人员费用项：基础工资、奖金、社保公积金、福利、销售提成、团建费。
- 日常管理要项：办公费、保险费、业务招待费、CRM 费用、差旅费、交通费。
- 运营费用：产品消耗、下午茶、活动物料、展台搭建、驻场费、签单礼、医生推荐费、现场品鉴、HCP 激励。

#### 3、数据链路

用画板或流程图表达数据从来源到看板的链路。

必须覆盖：

- 数据来源层：各业务系统、审批流、手工报表。
- 明细归集层：费用项、成本项、收入项如何进入统一明细。
- 模型加工层：ODS、DWD、ADS 的职责边界。
- 应用层：BI 看板、筛选项、核心指标。
- 人工依赖和风险点：哪些环节仍需财务或业务手工处理。

#### 4、数据建模

按 ODS、DWD、ADS 分层描述模型表。每张表要给出表名、中文名、字段清单和用途。

模板中的字段样式：

ODS/DWD 费用明细表：

| 字段名 | 中文名 |
| --- | --- |
| month_time | 月份 |
| shop_name | 门店/业务对象 |
| cost_category | 费用种类 |
| cost_item | 费用项 |
| remark | 备注 |
| cost_fee | 费用金额 |

ADS 单店/主题模型表：

| 字段名 | 中文名 |
| --- | --- |
| month_time | 月份 |
| shop_name | 门店/业务对象 |
| cost_category | 费用种类 |
| cost_item | 费用项 |
| cost_fee | 费用金额 |
| totalfee | 销售额 |
| product_cost | 产品成本 |

建模说明要写清：

- ODS：接入和保留来源明细，尽量少加工。
- DWD：统一清洗后的明细事实，完成口径映射、分类归集、基础规则处理。
- ADS：面向看板的汇总或宽表，补齐销售额、成本、毛利、利润等分析指标。

#### 5、数据看板

说明看板名称、筛选条件、核心视图和展示矩阵。

模板中的看板结构：

- 看板名称：{业务主题}单店模型看板。
- 筛选条件：门店筛选、月份筛选；必要时增加渠道、区域、费用分类。
- 主视图：按费用项为行，1-12 月为列，展示月度金额。
- 指标分层：运营利润、人员费用项、日常管理要项、运营费用。

| 运营利润 | 销售额 |  |  |  |  |  |  |  |  |  |  |  |  |
| 运营利润 | 产品成本 |  |  |  |  |  |  |  |  |  |  |  |  |
| 运营利润 | 毛利 |  |  |  |  |  |  |  |  |  |  |  |  |

## 新方案生成检查清单

生成数据方案前，优先向用户或上下文确认：

- 业务主题是什么，分析对象是门店、渠道、产品、客户、项目还是人员。
- 目标产物是什么，是模型表、BI 看板、报表、预警还是经营诊断。
- 统计周期是什么，常见为自然月、周、日或活动周期。
- 收入、成本、费用、数量等核心指标分别来自哪里。
- 哪些口径由业务确认，哪些由财务确认，哪些需要技术推导。
- 是否存在人工报表、审批流、第三方系统或截图信息。
- 是否有暂不纳入项、删除项、待确认项。
- 看板需要哪些筛选条件和核心展示方式。

## 输出风格

- 先业务，后技术；先目标，后链路；先口径，后模型。
- 保留未确认项，不要把风险写成确定事实。
- 表名用英文，中文名解释业务用途。
- 数据口径表要足够细，能让业务、财务、技术三方逐行确认。
- 建模部分不要只列字段，要说明每层模型承担的职责。
- 看板部分要能直接指导 BI 实现，包括筛选、指标分层和展示矩阵。
